A Practical Guide to Open-Source AI Coding Assistants

1 Introduction

1.1 What is a Coding Agent?

A coding agent is an AI system that can autonomously write, edit, debug, and reason about code. Unlike a simple chatbot that answers programming questions, a coding agent can:

  • Read and navigate your codebase
  • Execute shell commands and interpret results
  • Edit files across your project
  • Run tests and fix failures iteratively
  • Plan multi-step tasks and self-correct

You’ve probably heard of tools like Claude Code (Anthropic), GitHub Copilot (Microsoft), or Codex CLI (OpenAI). These are commercial, closed-source agents. But there’s a growing ecosystem of open-source alternatives that you can run locally, customize, and integrate into your own workflows — without sending proprietary code to third-party servers.

1.2 Why Should You Care?

If you work in a regulated industry (banking, healthcare, government), sending source code to external APIs may violate compliance policies. Open-source coding agents let you:

  • Keep code on-premise: Run models locally or on your own infrastructure
  • Customize behaviour: Add domain-specific tools, enforce coding standards, integrate with internal systems
  • Control costs: No per-token API fees for local models
  • Audit everything: Full transparency into what the model sees and does

1.3 What You’ll Learn

By the end of this guide, you will be able to:

  1. Understand the architecture of a coding agent
  2. Choose the right open-source model using benchmarks
  3. Set up a local coding agent with working code
  4. Extend it with custom tools
  5. (Bonus) Use API-based models when local isn’t enough

3 Architecture of a Coding Agent

Every coding agent, whether commercial or open-source, shares the same core architecture:

3.1 The Four Components

3.1.1 1. The LLM (the “Brain”)

The large language model that reasons about code. It receives context (your files, your request) and decides what to do next.

3.1.2 2. Tools (the “Hands”)

Functions the LLM can call to interact with the outside world:

  • File operations: read, write, edit files
  • Shell execution: run commands (tests, builds, git)
  • Search: find files, grep for patterns
  • Web access: fetch documentation, search for solutions

3.1.3 3. The Agent Loop

The orchestration logic that:

  1. Sends the user’s request + context to the LLM
  2. Parses the LLM’s response for tool calls
  3. Executes the tools
  4. Feeds results back to the LLM
  5. Repeats until the task is complete

3.1.4 4. Context Management

Code projects can be huge. The agent needs strategies to:

  • Select only relevant files to include in the prompt
  • Summarize large files
  • Maintain conversation history without exceeding token limits

4.2 Key Benchmarks for Coding Models

Not all benchmarks are equal. Here are the ones that actually matter for coding agents:

4.2.1 SWE-bench

The gold standard for evaluating coding agents. It tests whether an agent can autonomously resolve real GitHub issues from popular Python repositories.

  • What it measures: End-to-end bug fixing (read issue → find code → write patch → pass tests)
  • Where to check: swebench.com
  • What’s a good score: Top agents score 40-50% on SWE-bench Verified

4.2.2 BigCodeBench

Evaluates code generation across 1,140 practical programming tasks.

  • What it measures: Function-level code generation with real-world library usage
  • Where to check: bigcode-bench.github.io

4.2.3 HumanEval / HumanEval+

The classic benchmark: 164 hand-written Python programming problems.

  • What it measures: Basic code generation ability
  • Limitation: Too easy for modern models — most score 80%+
  • Where to check: evalplus.github.io

4.2.4 LiveCodeBench

Fresh competitive programming problems (post-training-cutoff) to avoid data contamination.

  • What it measures: Genuine reasoning and problem-solving
  • Where to check: livecodebench.github.io

Ollama exposes an OpenAI-compatible API on http://localhost:11434. This means any code written for the OpenAI API works with Ollama — just change the base URL. We’ll use this extensively in the hands-on sections.

5.3 Option B: LM Studio (GUI)

LM Studio is a desktop application for running local models. If you prefer a graphical interface over the terminal, this is for you.

5.3.1 Why LM Studio?

  • Visual model browser: Search, filter, and download models from Hugging Face directly in the app — no command line needed
  • One-click download: Pick a model, click download, and it handles quantization format selection (GGUF) automatically
  • Built-in chat UI: Test models in a ChatGPT-like interface before using them programmatically
  • Performance dashboard: See tokens/second, memory usage, and GPU utilization in real time
  • OpenAI-compatible server: Like Ollama, LM Studio can serve models via a local API

5.3.2 Install LM Studio

  1. Download from lmstudio.ai (available for macOS, Windows, Linux)
  2. Install and open the application
  3. You’ll see the Discover tab — this is the model browser

5.3.3 Downloading a Model

In the Discover tab:

  1. Search for qwen2.5-coder (or any model you want)
  2. You’ll see various quantization options (Q4_K_M, Q5_K_M, Q8_0, etc.)
    • Q4_K_M: Good balance of quality and size (recommended for most users)
    • Q8_0: Higher quality, needs more RAM
    • FP16: Full precision, needs significantly more RAM/VRAM
  3. Click the download button next to your chosen variant
  4. Wait for the download to complete
TipChoosing Quantization in LM Studio

LM Studio shows you the file size and estimated RAM usage for each quantization variant. A good rule of thumb: pick the largest quantization that fits in your available RAM with ~2GB headroom.

5.3.4 Using the Chat Interface

Once a model is downloaded:

  1. Go to the Chat tab (left sidebar)
  2. Select your model from the dropdown at the top
  3. Start chatting — use this to quickly test if a model is good at coding before integrating it into your workflow

6 Building a Minimal Coding Agent

Let’s build a coding agent from scratch. We’ll start simple and add features incrementally.

6.1 Version 1: A Chat Loop with Tool Calling

This is the simplest possible agent: it can read files, list directories, and run shell commands.

"""
minimal_agent.py — A bare-bones coding agent using a local LLM via Ollama.
"""

import json
import subprocess
import os
from openai import OpenAI

# ── Configuration ──────────────────────────────────────────────────────
MODEL = "qwen2.5-coder:7b"  # Change to your preferred model
client = OpenAI(
    base_url="http://localhost:11434/v1",  # Ollama's OpenAI-compatible endpoint
    api_key="ollama",                       # Ollama doesn't need a real key
)

# ── Tool Definitions ──────────────────────────────────────────────────
# These follow the OpenAI function-calling schema.
# The LLM reads these descriptions to decide when and how to call each tool.

TOOLS = [
    {
        "type": "function",
        "function": {
            "name": "read_file",
            "description": "Read the contents of a file at the given path.",
            "parameters": {
                "type": "object",
                "properties": {
                    "path": {
                        "type": "string",
                        "description": "Absolute or relative file path"
                    }
                },
                "required": ["path"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "list_directory",
            "description": "List files and folders in a directory.",
            "parameters": {
                "type": "object",
                "properties": {
                    "path": {
                        "type": "string",
                        "description": "Directory path (default: current directory)"
                    }
                },
                "required": []
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "run_command",
            "description": "Execute a shell command and return its output. Use for running tests, git commands, etc.",
            "parameters": {
                "type": "object",
                "properties": {
                    "command": {
                        "type": "string",
                        "description": "The shell command to execute"
                    }
                },
                "required": ["command"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "write_file",
            "description": "Write content to a file. Creates the file if it doesn't exist, overwrites if it does.",
            "parameters": {
                "type": "object",
                "properties": {
                    "path": {
                        "type": "string",
                        "description": "File path to write to"
                    },
                    "content": {
                        "type": "string",
                        "description": "Content to write"
                    }
                },
                "required": ["path", "content"]
            }
        }
    },
]


# ── Tool Implementations ─────────────────────────────────────────────

def read_file(path: str) -> str:
    """Read a file and return its contents (with basic safety check)."""
    path = os.path.abspath(path)
    if not os.path.isfile(path):
        return f"Error: File not found: {path}"
    try:
        with open(path, "r") as f:
            content = f.read()
        # Truncate very large files to avoid blowing up the context
        if len(content) > 10_000:
            return content[:10_000] + f"\n\n... [truncated, file is {len(content)} chars]"
        return content
    except Exception as e:
        return f"Error reading file: {e}"


def list_directory(path: str = ".") -> str:
    """List directory contents."""
    path = path or "."
    try:
        entries = os.listdir(path)
        entries.sort()
        result = []
        for entry in entries:
            full = os.path.join(path, entry)
            marker = "/" if os.path.isdir(full) else ""
            result.append(f"  {entry}{marker}")
        return f"Contents of {os.path.abspath(path)}:\n" + "\n".join(result)
    except Exception as e:
        return f"Error: {e}"


def run_command(command: str) -> str:
    """Execute a shell command with a timeout."""
    try:
        result = subprocess.run(
            command,
            shell=True,
            capture_output=True,
            text=True,
            timeout=30,  # Safety: don't let commands run forever
        )
        output = result.stdout + result.stderr
        if len(output) > 5_000:
            output = output[:5_000] + "\n... [truncated]"
        return output or "(no output)"
    except subprocess.TimeoutExpired:
        return "Error: Command timed out after 30 seconds"
    except Exception as e:
        return f"Error: {e}"


def write_file(path: str, content: str) -> str:
    """Write content to a file."""
    try:
        os.makedirs(os.path.dirname(path) or ".", exist_ok=True)
        with open(path, "w") as f:
            f.write(content)
        return f"Successfully wrote {len(content)} chars to {path}"
    except Exception as e:
        return f"Error writing file: {e}"


# Map tool names to their implementations
TOOL_MAP = {
    "read_file": read_file,
    "list_directory": list_directory,
    "run_command": run_command,
    "write_file": write_file,
}


# ── Agent Loop ────────────────────────────────────────────────────────

SYSTEM_PROMPT = """You are a helpful coding assistant with access to the local filesystem.
You can read files, list directories, write files, and run shell commands.
Always read relevant files before suggesting changes. Think step by step.
When editing code, show the complete modified file."""


def run_agent():
    """Main agent loop: chat with the user, call tools as needed."""
    messages = [{"role": "system", "content": SYSTEM_PROMPT}]
    print("Coding Agent (type 'quit' to exit)")
    print("=" * 40)

    while True:
        # Get user input
        user_input = input("\nYou: ").strip()
        if user_input.lower() in ("quit", "exit", "q"):
            print("Goodbye!")
            break
        if not user_input:
            continue

        messages.append({"role": "user", "content": user_input})

        # Agent loop: keep going until the LLM responds without tool calls
        while True:
            response = client.chat.completions.create(
                model=MODEL,
                messages=messages,
                tools=TOOLS,
                temperature=0.1,
            )

            choice = response.choices[0]
            message = choice.message

            # If the model wants to call tools, execute them
            if message.tool_calls:
                # Add the assistant's message (with tool calls) to history
                messages.append(message)

                for tool_call in message.tool_calls:
                    fn_name = tool_call.function.name
                    fn_args = json.loads(tool_call.function.arguments)

                    print(f"  [Tool: {fn_name}({fn_args})]")

                    # Execute the tool
                    if fn_name in TOOL_MAP:
                        result = TOOL_MAP[fn_name](**fn_args)
                    else:
                        result = f"Error: Unknown tool '{fn_name}'"

                    # Add tool result to message history
                    messages.append({
                        "role": "tool",
                        "tool_call_id": tool_call.id,
                        "content": result,
                    })

                # Continue the loop — the LLM will see the tool results
                continue

            else:
                # No tool calls — the LLM is done, print its response
                print(f"\nAgent: {message.content}")
                messages.append({"role": "assistant", "content": message.content})
                break


if __name__ == "__main__":
    run_agent()

6.1.1 How It Works

  1. The user types a request (e.g., “read main.py and add error handling”)
  2. The LLM receives the request + tool definitions
  3. The LLM decides to call read_file("main.py")
  4. The agent executes the tool and feeds the file contents back to the LLM
  5. The LLM now has the file contents and can reason about changes
  6. The LLM calls write_file("main.py", "...") with the updated code
  7. The agent writes the file and confirms
  8. The LLM responds to the user that the task is done

7 Adding Intelligence: Better Context and Search

The minimal agent works, but it’s naive — it has no way to search a codebase. Let’s add grep and glob tools, plus smarter context management.

7.1 Version 2: With Code Search

"""
Additional tools to make the agent smarter at navigating code.
Add these to the TOOLS list and TOOL_MAP from Version 1.
"""

import glob as glob_module


def search_files(pattern: str, path: str = ".") -> str:
    """Search for files matching a glob pattern."""
    try:
        matches = glob_module.glob(pattern, root_dir=path, recursive=True)
        matches.sort()
        if not matches:
            return f"No files matching '{pattern}'"
        result = matches[:50]  # Limit results
        output = f"Files matching '{pattern}':\n"
        output += "\n".join(f"  {m}" for m in result)
        if len(matches) > 50:
            output += f"\n  ... and {len(matches) - 50} more"
        return output
    except Exception as e:
        return f"Error: {e}"


def grep_code(pattern: str, path: str = ".", file_type: str = "") -> str:
    """Search file contents using grep."""
    cmd = f'grep -rn "{pattern}" "{path}"'
    if file_type:
        cmd += f' --include="*.{file_type}"'
    cmd += " | head -30"  # Limit output
    return run_command(cmd)


# Add to TOOLS list:
SEARCH_TOOLS = [
    {
        "type": "function",
        "function": {

            "parameters": {
                "type": "object",
                "properties": {
                    "pattern": {"type": "string", "description": "Glob pattern"},
                    "path": {"type": "string", "description": "Root directory (default: '.')"}
                },
                "required": ["pattern"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "grep_code",
            "description": "Search for a text pattern in file contents. Returns matching lines with file paths and line numbers.",
            "parameters": {
                "type": "object",
                "properties": {
                    "pattern": {"type": "string", "description": "Search pattern (regex supported)"},

12 Security Considerations

When building a coding agent — especially one that can execute shell commands — security matters.

12.1 Key Principles

  1. Sandboxing: Never run an agent with root privileges. Use Docker containers or restricted user accounts.

  2. Path restrictions: Limit file access to the project directory:

import os

ALLOWED_ROOT = os.path.abspath("./my-project")

def safe_read_file(path: str) -> str:
    """Read a file, but only within the allowed directory."""
    abs_path = os.path.abspath(path)
    if not abs_path.startswith(ALLOWED_ROOT):
        return f"Error: Access denied. Path must be within {ALLOWED_ROOT}"
    # ... proceed with reading
  1. Command allowlisting: Don’t let the agent run arbitrary commands in production:
ALLOWED_COMMANDS = ["python", "pytest", "git", "ls", "cat", "grep"]

def safe_run_command(command: str) -> str:
    """Only allow pre-approved commands."""
    first_word = command.split()[0]
    if first_word not in ALLOWED_COMMANDS:
        return f"Error: Command '{first_word}' is not allowed."
    # ... proceed with execution

  1. Token/cost limits: Set maximum tokens per request and total spend limits when using APIs.

  2. Human-in-the-loop: For destructive actions (file deletion, git push), always ask for confirmation.

# Introduction

## What is a Coding Agent?

A **coding agent** is an AI system that can autonomously write, edit, debug, and reason about code. Unlike a simple chatbot that answers programming questions, a coding agent can:

- Read and navigate your codebase
- Execute shell commands and interpret results
- Edit files across your project
- Run tests and fix failures iteratively
- Plan multi-step tasks and self-correct

You've probably heard of tools like **Claude Code** (Anthropic), **GitHub Copilot** (Microsoft), or **Codex CLI** (OpenAI). These are commercial, closed-source agents. But there's a growing ecosystem of **open-source alternatives** that you can run locally, customize, and integrate into your own workflows — without sending proprietary code to third-party servers.

## Why Should You Care?

If you work in a regulated industry (banking, healthcare, government), sending source code to external APIs may violate compliance policies. Open-source coding agents let you:

- **Keep code on-premise**: Run models locally or on your own infrastructure
- **Customize behaviour**: Add domain-specific tools, enforce coding standards, integrate with internal systems
- **Control costs**: No per-token API fees for local models
- **Audit everything**: Full transparency into what the model sees and does

# Architecture of a Coding Agent

Every coding agent, whether commercial or open-source, shares the same core architecture:

```{mermaid}
flowchart LR
    A[User Prompt] --> B[LLM Brain]
    B --> C{Tool Selection}
    C --> D[Read Files]
    C --> E[Edit Files]
    C --> F[Run Commands]
    C --> G[Search Code]
    D --> B
    E --> B
    F --> B
    G --> B
    B --> H[Response to User]
```

## The Four Components

### 1. The LLM (the "Brain")

The large language model that reasons about code. It receives context (your files, your request) and decides what to do next.

### 2. Tools (the "Hands")

Functions the LLM can call to interact with the outside world:

- **File operations**: read, write, edit files
- **Shell execution**: run commands (tests, builds, git)
- **Search**: find files, grep for patterns
- **Web access**: fetch documentation, search for solutions

### 3. The Agent Loop

The orchestration logic that:

1. Sends the user's request + context to the LLM
2. Parses the LLM's response for tool calls
3. Executes the tools
4. Feeds results back to the LLM
5. Repeats until the task is complete

### 4. Context Management

Code projects can be huge. The agent needs strategies to:

- Select only relevant files to include in the prompt
- Summarize large files
- Maintain conversation history without exceeding token limits

# Choosing the Right Model

## Where to Find Models

The go-to resource is [Hugging Face](https://huggingface.co), the largest open-source ML platform. Here's how to navigate it:

### Hugging Face Model Hub

Go to [huggingface.co/models](https://huggingface.co/models) and filter by:

- **Task**: `Text Generation`
- **Library**: `transformers`, `gguf` (for local inference)
- **Sort by**: `Trending` or `Most Downloads`

### What to Look For in a Model Card

Every model on Hugging Face has a **model card** — a README with key information. Here's what matters for coding agents:

| Field | What to check |
|-------|--------------|
| **Parameters** | 7B runs on a laptop, 13B-34B needs a good GPU, 70B+ needs multi-GPU |
| **Context window** | Minimum 8K tokens, ideally 32K+ for code |
| **License** | `Apache 2.0` or `MIT` for commercial use. Watch out for `Llama Community License` restrictions |
| **Training data** | Was it trained on code? Look for mentions of GitHub, The Stack, CodeSearchNet |
| **Quantization** | GGUF/GPTQ versions available? These run faster with less RAM |

## Key Benchmarks for Coding Models

Not all benchmarks are equal. Here are the ones that actually matter for coding agents:

### SWE-bench

The gold standard for evaluating coding agents. It tests whether an agent can autonomously resolve real GitHub issues from popular Python repositories.

- **What it measures**: End-to-end bug fixing (read issue → find code → write patch → pass tests)
- **Where to check**: [swebench.com](https://www.swebench.com)
- **What's a good score**: Top agents score 40-50% on SWE-bench Verified

### BigCodeBench

Evaluates code generation across 1,140 practical programming tasks.

- **What it measures**: Function-level code generation with real-world library usage
- **Where to check**: [bigcode-bench.github.io](https://bigcode-bench.github.io)

### HumanEval / HumanEval+

The classic benchmark: 164 hand-written Python programming problems.

- **What it measures**: Basic code generation ability
- **Limitation**: Too easy for modern models — most score 80%+
- **Where to check**: [evalplus.github.io](https://evalplus.github.io/leaderboard.html)

### LiveCodeBench

Fresh competitive programming problems (post-training-cutoff) to avoid data contamination.

- **What it measures**: Genuine reasoning and problem-solving
- **Where to check**: [livecodebench.github.io](https://livecodebench.github.io)

```{python}
        }
    },
    {
        "type": "function",
        "function": {
            "name": "write_file",
            "description": "Write content to a file. Creates the file if it doesn't exist, overwrites if it does.",
            "parameters": {
                "type": "object",
                "properties": {
                    "path": {
                        "type": "string",
                        "description": "File path to write to"
                    },
                    "content": {
                        "type": "string",
                        "description": "Content to write"
                    }
                },
                "required": ["path", "content"]
            }
        }
    },
]


# ── Tool Implementations ─────────────────────────────────────────────

def read_file(path: str) -> str:
    """Read a file and return its contents (with basic safety check)."""
    path = os.path.abspath(path)
    if not os.path.isfile(path):
        return f"Error: File not found: {path}"
    try:
        with open(path, "r") as f:
            content = f.read()
        # Truncate very large files to avoid blowing up the context
        if len(content) > 10_000:
            return content[:10_000] + f"\n\n... [truncated, file is {len(content)} chars]"
        return content
    except Exception as e:
        return f"Error reading file: {e}"


def list_directory(path: str = ".") -> str:
    """List directory contents."""
    path = path or "."
    try:
        entries = os.listdir(path)
        entries.sort()
        result = []
        for entry in entries:
            full = os.path.join(path, entry)
            marker = "/" if os.path.isdir(full) else ""
            result.append(f"  {entry}{marker}")
        return f"Contents of {os.path.abspath(path)}:\n" + "\n".join(result)
    except Exception as e:
        return f"Error: {e}"


def run_command(command: str) -> str:
    """Execute a shell command with a timeout."""
    try:
        result = subprocess.run(
            command,
            shell=True,
            capture_output=True,
            text=True,
            timeout=30,  # Safety: don't let commands run forever
        )
        output = result.stdout + result.stderr
        if len(output) > 5_000:
            output = output[:5_000] + "\n... [truncated]"
        return output or "(no output)"
    except subprocess.TimeoutExpired:
        return "Error: Command timed out after 30 seconds"
    except Exception as e:
        return f"Error: {e}"


def write_file(path: str, content: str) -> str:
    """Write content to a file."""
    try:
        os.makedirs(os.path.dirname(path) or ".", exist_ok=True)
        with open(path, "w") as f:
            f.write(content)
        return f"Successfully wrote {len(content)} chars to {path}"
    except Exception as e:
        return f"Error writing file: {e}"


# Map tool names to their implementations
TOOL_MAP = {
    "read_file": read_file,
    "list_directory": list_directory,
    "run_command": run_command,
    "write_file": write_file,
}


# ── Agent Loop ────────────────────────────────────────────────────────

SYSTEM_PROMPT = """You are a helpful coding assistant with access to the local filesystem.
You can read files, list directories, write files, and run shell commands.
Always read relevant files before suggesting changes. Think step by step.
When editing code, show the complete modified file."""


def run_agent():
    """Main agent loop: chat with the user, call tools as needed."""
    messages = [{"role": "system", "content": SYSTEM_PROMPT}]
    print("Coding Agent (type 'quit' to exit)")
    print("=" * 40)

    while True:
        # Get user input
        user_input = input("\nYou: ").strip()
        if user_input.lower() in ("quit", "exit", "q"):
            print("Goodbye!")
            break
        if not user_input:
            continue

        messages.append({"role": "user", "content": user_input})

        # Agent loop: keep going until the LLM responds without tool calls
        while True:
            response = client.chat.completions.create(
                model=MODEL,
                messages=messages,
                tools=TOOLS,
                temperature=0.1,
            )

            choice = response.choices[0]
            message = choice.message

            # If the model wants to call tools, execute them
            if message.tool_calls:
                # Add the assistant's message (with tool calls) to history
                messages.append(message)

                for tool_call in message.tool_calls:
                    fn_name = tool_call.function.name
                    fn_args = json.loads(tool_call.function.arguments)

                    print(f"  [Tool: {fn_name}({fn_args})]")

                    # Execute the tool
                    if fn_name in TOOL_MAP:
                        result = TOOL_MAP[fn_name](**fn_args)
                    else:
                        result = f"Error: Unknown tool '{fn_name}'"

                    # Add tool result to message history
                    messages.append({
                        "role": "tool",
                        "tool_call_id": tool_call.id,
                        "content": result,
                    })

                # Continue the loop — the LLM will see the tool results
                continue

            else:
                # No tool calls — the LLM is done, print its response
                print(f"\nAgent: {message.content}")
                messages.append({"role": "assistant", "content": message.content})
                break


if __name__ == "__main__":
    run_agent()
```

### How It Works

1. The user types a request (e.g., "read main.py and add error handling")
2. The LLM receives the request + tool definitions
3. The LLM decides to call `read_file("main.py")`
4. The agent executes the tool and feeds the file contents back to the LLM
5. The LLM now has the file contents and can reason about changes
6. The LLM calls `write_file("main.py", "...")` with the updated code
7. The agent writes the file and confirms
8. The LLM responds to the user that the task is done

# Adding Intelligence: Better Context and Search

The minimal agent works, but it's naive — it has no way to search a codebase. Let's add **grep** and **glob** tools, plus smarter context management.

## Version 2: With Code Search

```{python}
#| label: search-tools
"""
Additional tools to make the agent smarter at navigating code.
Add these to the TOOLS list and TOOL_MAP from Version 1.
"""

import glob as glob_module


def search_files(pattern: str, path: str = ".") -> str:
    """Search for files matching a glob pattern."""
    try:
        matches = glob_module.glob(pattern, root_dir=path, recursive=True)
        matches.sort()
        if not matches:
            return f"No files matching '{pattern}'"
        result = matches[:50]  # Limit results
        output = f"Files matching '{pattern}':\n"
        output += "\n".join(f"  {m}" for m in result)
        if len(matches) > 50:
            output += f"\n  ... and {len(matches) - 50} more"
        return output
    except Exception as e:
        return f"Error: {e}"


def grep_code(pattern: str, path: str = ".", file_type: str = "") -> str:
    """Search file contents using grep."""
    cmd = f'grep -rn "{pattern}" "{path}"'
    if file_type:
        cmd += f' --include="*.{file_type}"'
    cmd += " | head -30"  # Limit output
    return run_command(cmd)
```

# Using OpenCode: The Open-Source Claude Code {#sec-opencode}

[OpenCode](https://opencode.ai) is a terminal-based coding agent that feels very similar to **Claude Code** or **Codex CLI** — but it's fully open-source (MIT licensed) and works with any LLM provider, including local models.

If you've used Claude Code and want something similar that works with open-weight models or your own API keys, OpenCode is the closest match.

## What Makes OpenCode Different?

| Feature | OpenCode | Aider | SWE-agent |
|---------|----------|-------|-----------|
| **UX** | Rich terminal UI (like Claude Code) | Simple chat | Autonomous (no interaction) |
| **Interaction style** | Interactive, iterative | Interactive | Fully autonomous |
| **File editing** | Inline diffs with approval | Git-based patches | Patch files |
| **LSP support** | Yes (diagnostics, go-to-definition) | No | No |
| **Session management** | Built-in (resume previous sessions) | Git history | Logs |
| **MCP support** | Yes | No | No |

## Using OpenCode

### Basic Usage

```{bash}

opencode
```

This opens an interactive terminal UI. You can:

- Type natural language requests ("add input validation to the login function")
- See file diffs before they're applied
- Approve or reject changes

### Key Features in Practice

**File context**: OpenCode automatically indexes your project and pulls in relevant files based on your query — you don't need to manually add files like in Aider.

**Diff review**: When OpenCode proposes a change, you see a side-by-side diff and can accept, reject, or ask for modifications. This is similar to Claude Code's permission model.

**Session persistence**: If you close OpenCode and reopen it, you can resume your previous session with full context.

**LSP integration**: OpenCode can connect to language servers for real-time diagnostics. If your change introduces a type error or syntax issue, it catches it immediately.

|----------|---------------|
| Simple code generation, autocomplete | Local model (7B-32B) |
| Complex refactoring, multi-file changes | API model or large local (70B+) |
| Regulated environment, no external access | Local model only |

```{python}
        {
            "name": "read_file",
```

# Security Considerations

When building a coding agent — especially one that can execute shell commands — security matters.

## Key Principles

1. **Sandboxing**: Never run an agent with root privileges. Use Docker containers or restricted user accounts.

2. **Path restrictions**: Limit file access to the project directory:

```{python}
#| label: path-safety
import os

ALLOWED_ROOT = os.path.abspath("./my-project")

def safe_read_file(path: str) -> str:
    """Read a file, but only within the allowed directory."""
    abs_path = os.path.abspath(path)
    if not abs_path.startswith(ALLOWED_ROOT):
        return f"Error: Access denied. Path must be within {ALLOWED_ROOT}"
    # ... proceed with reading
```

3. **Command allowlisting**: Don't let the agent run arbitrary commands in production:

```{python}
#| label: command-safety
ALLOWED_COMMANDS = ["python", "pytest", "git", "ls", "cat", "grep"]

def safe_run_command(command: str) -> str:
    """Only allow pre-approved commands."""
    first_word = command.split()[0]
    if first_word not in ALLOWED_COMMANDS:
        return f"Error: Command '{first_word}' is not allowed."
    # ... proceed with execution
```

4. **Token/cost limits**: Set maximum tokens per request and total spend limits when using APIs.

5. **Human-in-the-loop**: For destructive actions (file deletion, git push), always ask for confirmation.
